# 05. Подход к дизайну

## Принцип «максимальной нативности»

Сознательное продуктовое решение — не строить кросс-платформенную дизайн-систему, а следовать нативным рекомендациям каждой платформы. Цель: приложение должно ощущаться так, как если бы оно было встроено в систему, а не «прикручено сверху».

| Платформа | Гайдлайны | Особенности |
|-----------|-----------|-------------|
| iOS | Apple Human Interface Guidelines (HIG) | Полная адаптация под iOS 26+ (Liquid Glass во всех уместных элементах) и под iOS 18− (родные системные компоненты без Liquid Glass). |
| Android | Material Design 3 Expressive | Динамические темы, акцентные цвета пользователя сильно меняют внешний вид. |
| Web | shadcn/ui поверх Tailwind CSS | Адаптивная вёрстка под десктоп и мобильное устройство; общий визуальный язык лендинга, веб-приложения расписания и веб-сервиса администрирования. |

## Источник вдохновения

Эталоном по работе с нативностью в продакт-дизайне считаем Telegram. Подход: «приложение из коробки», как если бы оно входило в стандартную поставку операционной системы.

## Figma-файлы

Для каждой платформы поддерживается **отдельный Figma-файл**. Все файлы хранятся в единой Figma-организации проекта. Это сознательная декомпозиция: попытка вести общий мульти-платформенный файл означала бы «общий знаменатель» между HIG и Material, который не даёт нативного результата ни на одной из платформ.

## Что НЕ делаем

- Не используем кросс-платформенные UI-фреймворки (Flutter, React Native, Compose Multiplatform) для основного UI.
- Не делаем «универсальную» дизайн-систему поверх HIG/Material/shadcn.
- Не используем sans-serif кастомные шрифты в основном UI — везде системные.
