# Backend: то, что мы знаем со стороны клиентов

Бэкенд разрабатывается соавтором (Максим, CTO). В рамках ВКР автора рассматривается **только в объёме интеграционного контракта**. Сам бэк описывается в **отдельной ВКР** Максима.

## 0. Два контура бэкенда

В экосистеме «ВКурсе» **два независимых бэкенд-контура**:

| Контур | Адрес | Назначение | Кем разработан | Статус |
|---|---|---|---|---|
| **Парсинг (legacy)** | `api.vcourse.app` | Просмотр расписаний 6 омских вузов через парсинг их официальных сайтов | Максим (вне его ВКР) | В продакшене с 2025 |
| **Платформа (SaaS)** | `platform.vcourse.app` | Собственный сервис управления расписанием (для web-admin) | Максим (это его ВКР) | MVP в разработке |

**Целевая картина к защите:** оба контура сосуществуют. Парсинг отдаёт расписания тех вузов, что мы пока не подключили к собственному сервису. Платформа отдаёт расписания тех вузов, что уже завели расписание у нас. Перспектива — встраивание парсингового слоя в платформу за единым адресом `api.vcourse.app`, чтобы клиенты не знали о различии источников.

## 1. Парсинговый контур (`api.vcourse.app`)

Используется **всеми тремя клиентами просмотра расписания**: iOS, Android, Web-public.

### Что известно со стороны клиентов

- **Базовый URL:** `https://api.vcourse.app/`.
- **Формат:** REST + JSON.
- **Заголовки всех запросов (одинаковы для iOS, Android и Web):**
  - `Authorization: Bearer 91ed68b2-a5bd-41a3-b0cb-df06a933e298` — единый клиентский ключ платформы. Идентифицирует не пользователя, а факт легитимного клиента «ВКурсе». **На вебе** этот заголовок подставляется на стороне Next.js Route Handler, а не передаётся в браузер (через env `VC_AUTH_TOKEN`).
  - `X-API-Version: 2` — версия контракта API.
  - `X-Installation-ID` — UUID конкретной установки клиента (генерируется при первом запуске, хранится локально: UserDefaults на iOS через App Group, SharedPreferences на Android, не применимо для веба).
  - `User-Agent` — собирается клиентом из имени, версии, билда, ОС, модели устройства. Пример iOS: `VCourse/0.2.3.1 (iOS/18.5; iPhone15,3) Darwin/24.0.0`. Пример Android: `VCourse/0.1.5.10 (Android/14; Samsung SM-G990B)`.
- **Известные эндпойнты:**
  - `GET /universities` — список вузов (на Web принудительно добавляется `?available=true`).
  - `GET /timetable/{universityId}` — сетка пар вуза (slots: start/end).
  - `GET /students-groups/{universityId}`, `GET /tutors/{universityId}`, `GET /classrooms/{universityId}`, `GET /buildings/{universityId}` — справочники.
  - `GET /schedule/{universityId}/{group|tutor|classroom}/{entityId}?since=&until=` — расписание сущности.
  - `GET /schedule/.../count?since=&until=` — количество пар (используется виджетом iOS).
- **Окно загрузки расписания:** клиенты запрашивают `±1 год` от текущей даты, формат дат `yyyy-MM-dd` (UTC).
- **Логотипы вузов:** грузятся отдельным URL (значение поля `logo_url` в ответе `/universities`); на Web проксируются через `/api/static/[...path]` с 24-часовой ревалидацией и immutable browser-cache.

### Модель пользователя

Парсинговый контур **не имеет персональной аутентификации**: клиенты iOS/Android/Web-public ходят как анонимные читатели, идентифицируются только через клиентский ключ + `X-Installation-ID`. Экраны входа в мобильных приложениях в MVP **не планируются** даже после интеграции с собственным сервисом — мобильные клиенты по архитектуре остаются «просмотрщиками».

### Аналитика поверх парсингового контура

Grafana поверх метрик бэка. Подсчёт уникальных и новых пользователей — через `X-Installation-ID`. Распределение по вузам и платформам собирается агрегацией заголовков запросов.

## 2. Платформенный контур (`platform.vcourse.app`)

Новая микросервисная архитектура — **предмет ВКР Максима**. В нашей работе описывается **только как контекст** для web-admin (раздел 3.3.2 ВКР).

### Технологический стек

- **Язык:** Java 25.
- **Фреймворк:** Spring Boot 3.5.
- **Сборщик:** Maven.
- **СУБД:** PostgreSQL.
- **Кеш:** Redis.
- **Брокер сообщений:** RabbitMQ.
- **Контейнеризация:** Docker (каждый микросервис — отдельный образ).
- **Оркестрация (prod):** Kubernetes (предполагается высокодоступный кластер).
- **Dev-окружение:** Docker Compose + Nginx как reverse-proxy.

### API

- **Тип API:** REST.
- **Спецификация:** OpenAPI + Swagger UI на каждом микросервисе. На момент написания работы публично доступен только сервис аутентификации: `https://dev.platform.vcourse.app/swagger-ui/index.html`. Остальные сервисы (включая Core) — позже.
- **Версионирование:** через HTTP-заголовок `X-API-Version` (тот же подход, что и в парсинговом контуре).
- **Пагинация:**
  - **Page-Based** (`page` + `size`) — для списков сущностей (группы, преподаватели, аудитории и т.д.).
  - **Time-Based** (`since` + `until`) — для данных расписания (совпадает с парсинговым контуром).

### Аутентификация и авторизация

- **Схема:** OAuth 2.0 **Resource Owner Password Credentials Flow** (ROPC). Клиент шлёт логин и пароль, получает пару JWT-токенов.
- **Токены:**
  - `access-token` — короткоживущий, **TTL 5 минут**, используется для авторизации защищённых эндпойнтов;
  - `refresh-token` — долгоживущий, **TTL 90 дней**, используется для обновления access после истечения.
  - Алгоритм подписи — RS256, есть `kid` в заголовке.
- **Структура access-token (claims):**
  ```json
  {
    "jti":  "...",                       // уникальный идентификатор токена
    "sub":  "<userId UUID>",
    "iss":  "https://platform.vcourse.app",
    "email": "...",
    "last_name": "...", "middle_name": "...", "first_name": "...",
    "iat":  1779125996, "exp": 1779126296,
    "authorities": ["ROLE_USER"],        // глобальные права на платформе
    "orgs": {                            // права в рамках организаций
      "<orgId UUID>": ["ROLE_ADMIN"]
    }
  }
  ```
- **Ролевая модель:**
  - Глобальные роли: `ROLE_USER`, `ROLE_ADMIN`.
  - Per-org роли: внутри `orgs` для каждой организации список ролей. На этапе MVP — один пользователь привязан **только к одной организации**.
- **Хранение токенов на клиенте:** в безопасном хранилище браузера (отдаём решение клиентской стороне). Для web-admin принято: **JWT-пара хранится на клиенте** (см. [clients/web-admin.md](clients/web-admin.md), раздел 4). CORS на бэкенде конфигурируется под фактическую схему хранения.

### Доменная модель и валидация

- **ER-диаграмма / структура данных:** в работе у Максима на текущей неделе. До получения — список доменных типов берём из web-admin (`UniversityData`, `Faculty`, `Department`, `Building`, `Classroom`, `StudentGroup`, `Tutor`, `Employee`, `Stream`, `Discipline`, `User`, `LessonAtom`, `Lesson`, `ChangelogEntry`).
- **Валидация коллизий:** на уровне **бизнес-логики** (прикладной слой Spring), базовые ограничения целостности — через constraints в PostgreSQL.
- **Аудит-логи:** автоматически через **Hibernate Envers** — каждая ауд-таблица генерируется рядом со своей основной.

### Инфраструктура и наблюдаемость

- **Метрики:** Spring Boot Actuator → Prometheus → Grafana.
- **Деплой prod:** Kubernetes (детали — за рамками нашей ВКР).
- **CI/CD:** GitHub Actions. Пайплайн:
  1. Проверка сборки артефактов;
  2. Сборка Docker-образа и публикация в **GHCR** (GitHub Container Registry);
  3. Миграции БД исполняются микросервисами при старте через **Liquibase** (манифесты — часть исходников).
- **Тестирование:** на момент написания работы **тестов нет**. План — покрыть микросервисы юнит- и интеграционными.

### Что в платформенном контуре пока не реализовано

- **Парсинг расписаний** — отсутствует (отдельно живёт в парсинговом контуре).
- **Реальное время / push** — отсутствует. Доставка обновлений в клиенты — pull при следующем запросе расписания.
- **ETag / If-Modified-Since** — отсутствует.
