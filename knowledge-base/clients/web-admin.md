- **Редактор расписания — недописан**:

- **Полированные мелочи** по всему сервису: кое-где не хватает кнопок, локализаций, состояний пустого экрана — это правится точечно к релизу.

Это нормальное состояние недописанной части продукта. Ниже описывается **целевое состояние, реализованное к защите** — оно и идёт в ВКР.

## 2. Общая характеристика

- **Назначение:** SaaS-сервис для администрации учебного заведения — управление справочниками вуза, ведение расписания, управление сотрудниками. Заменяет собой устаревшие настольные решения вроде «Галактика РУЗ» и «ММИС Лаб».
- **Размещение:** на том же домене `vcourse.app`, в роут-группе `src/app/[locale]/(service)/`. Изоляция от публичной части — через отдельный layout и middleware-проверку доступа.
- **Точки входа:**
  - `/[locale]/authorization` — экран входа;
  - `/[locale]/onboarding` — первичная настройка вуза (после первой регистрации);
  - `/[locale]/workspace/*` — рабочий стол сотрудника.
- **Аудитория:** два уровня пользователей внутри одного учебного заведения:
  - **администратор** — добавление/удаление сотрудников, изменение данных заведения (название, логотип, домен), плюс всё, что может сотрудник;
  - **сотрудник** (методист / диспетчер) — работа со справочниками и расписанием.

## 3. Маршрутизация и страницы

### 3.1. Карта маршрутов

```
[locale]/
├── authorization/                          — вход
├── onboarding/                             — первичная настройка вуза
└── workspace/
    ├── /                                   — редирект на /workspace/dashboard
    ├── dashboard/                          — главный экран сотрудника
    ├── schedule/                           — редактор расписания (специальный лейаут с правым сайдбаром)

Маршруты сгруппированы внутри `workspace` через подгруппы `(main)` (dashboard / schedule / employees) и `(data)` (справочники) — это чисто организационное разделение, на URL не влияет.

### 3.2. `WorkspaceLayout`

`src/app/[locale]/(service)/workspace/layout.tsx` — общий лейаут, содержит:

- **Левый сайдбар** (`WorkspaceSidebar`) — навигация по разделам, сворачиваемый, состояние в `localStorage` (`workspace-sidebar-collapsed`). Карта пунктов — в `sidebar/navigation.ts` (`WORKSPACE_NAVIGATION` — две группы: `main` и `data`).
- **Шапка** (`WorkspaceHeader`) — глобальные действия, заголовок, поиск.
- **Правый сайдбар** — появляется только на маршруте `/workspace/schedule` (`ScheduleRightSidebar`). Тоже сворачиваемый, своё состояние в `localStorage`.
- **Адаптивность:** на `<768px` оба сайдбара принудительно сворачиваются.

Для маршрута `/workspace/schedule` вся ветка оборачивается в `<ScheduleProvider>` (см. раздел 7) — это даёт детям доступ к контексту редактора через `useSchedule()`.

## 4. Авторизация

**Целевая модель** (к защите) — определяется контрактом платформенного бэка (см. [backend-contract.md, раздел 2](../backend-contract.md#2-платформенный-контур-platformvcourseapp)):

- **Схема:** OAuth 2.0 **Resource Owner Password Credentials Flow**. Клиент шлёт email + пароль, бэк возвращает пару JWT.
- **Токены:**
  - `access-token` — TTL **5 минут**, кладётся в заголовок `Authorization: Bearer ...` к каждому защищённому запросу;
  - `refresh-token` — TTL **90 дней**, используется для обмена на новую пару при истечении access.
  - Алгоритм подписи RS256, claims: `sub`, `email`, ФИО, `authorities` (глобальные роли), `orgs` (per-org-роли).
- **Хранение токенов на клиенте:** в безопасном хранилище браузера. CORS на стороне бэкенда сконфигурирован под фактическую схему хранения.
- **Защита роутов:** Next.js middleware на маршруте `/[locale]/workspace/**` проверяет наличие действующего access-token, при отсутствии — редирект на `/[locale]/authorization`.
- **Обновление токенов:** при получении 401 от защищённого эндпойнта клиент пытается обменять refresh на новую пару; при неудаче — выходит в форму логина.
- **Rate-limit** на эндпойнт логина — на стороне бэкенда.
- **Cooldown на «забыл пароль»** — 60 секунд между запросами, состояние храним в `localStorage` (`auth_forgot_password_expiry`) — уже реализован в `LoginForm`.

**Форма входа** (`authorization/login-form/login-form.tsx`):
- email + пароль с валидацией через хук `useValidation` (правила `onlyLatin`, `invalidEmail`);
- чекбокс «Запомнить меня» (продлевает срок жизни cookie);
- toast-уведомления через `sonner` для ошибок и состояний;
- индикатор отправки через `Loader2Icon` + блокировка повторного submit;
- ссылка на политику конфиденциальности.

### 4.1. Ролевая модель

После авторизации в claims access-token приходит блок `orgs: { "<orgId>": ["ROLE_ADMIN"] }`. В MVP пользователь привязан только к одной организации; роль внутри неё — `ROLE_ADMIN` (администратор учебного заведения) или базовая (сотрудник без админ-прав).

На клиенте:

- **Условный рендеринг** пунктов сайдбара: `employees` в группе `main` показывается только при наличии `ROLE_ADMIN` в текущей организации;
- **Защита роутов** на уровне layout — переход на `/workspace/employees` без прав возвращает 404 или redirect на `/workspace/dashboard`;
- **UI-роль-чекап** — компонент-обёртка `<RequireAdmin>` оборачивает чувствительные действия (удаление сотрудника, смена реквизитов вуза).

## 6. Слой данных

### 6.1. Целевая схема

После подключения бэкенда:

- **Запросы** идут через те же Next.js Route Handlers, что и публичная часть (`/api/vcourse/...`), но с **server-side проверкой cookie** в middleware и серверным `ProfileAPI.load()` для получения роли;
- **Чтение** доменных сущностей — серверные компоненты (RSC) подгружают данные сразу в JSX (`async function Page()`). Это сокращает «двойной ходом» (server + client refetch) и ускоряет первый рендер таблиц.
- **Мутации** — через **Server Actions** (Next.js Form Actions). Это даёт прогрессивный enhancement (форма работает без JS) и встроенную ревалидацию кеша через `revalidatePath` / `revalidateTag`.

### 6.2. Заглушки и помощники сейчас

Сейчас в коде используется единый `MOCK_UNIVERSITY_DATA` (`src/core/service/mocks/university-data.ts`) — собирает граф сущностей из JSON-файлов и навешивает на сущности предупреждения через `get*Warnings()` (`utils/get-building-warnings.ts`, `get-department-warnings.ts`, `get-faculty-warnings.ts`, `get-tutor-warnings.ts`). Эти валидаторы — **готовая бизнес-логика**: проверки на пустые связи, недостающие поля и т.п. При переезде на бэкенд логика останется на сервере, на клиенте — индикация результата.

## 7. Редактор расписания

Главный и самый сложный экран сервиса — `/workspace/schedule`. Архитектура построена вокруг **атомарной модели уроков** + **редакторского состояния с историей действий**.

### 7.1. `LessonAtom` — атомарная модель

`LessonAtom` (`src/core/service/domain/lesson-atom.ts`) — элементарная неделимая единица расписания, описывающая **одну ячейку времени для одной сущности**:

- `date`, `slotIndex` — когда и в какую пару;
- ссылки на участников (`primaryEntities`, `secondaryEntity` — преподаватель, группа, аудитория, подгруппа);
- `subject`, `lessonType` (лекция / семинар / лаб / ...);
- `subgroupInfo` — индикация подгруппового деления.

Один «логический урок» (то, что пользователь видит как одну карточку) **состоит из набора атомов** — например, «лекция Иванова И.И. для групп ИВТ-21, ИВТ-22 в ауд. 305» = 3 атома: «для группы ИВТ-21», «для группы ИВТ-22», «для аудитории 305». Атомы хранятся плоским массивом, агрегация в `Lesson` (для отображения в сетке) — на клиенте через `aggregateAtoms(atoms, viewpoint, lookup)` (`network/lesson-atom-mapper.ts`).

Это позволяет:

- **Простую дельту** при редактировании: добавить/удалить/заменить набор атомов — а не пересобирать «весь урок»;
- **Множественные точки зрения**: одни и те же атомы агрегируются по-разному для группы (видит «свой урок»), для преподавателя (видит «свой урок», даже если он ведёт двум потокам параллельно), для аудитории (видит занятость);
- **Состояния «новый / отредактированный / без изменений»**: каждый атом помечается своим origin (snapshot, dirty, new), визуальный индикатор на карточке — на основе этого origin.

### 7.2. `ScheduleProvider` — контекст редактора

Корневой компонент `<ScheduleProvider>` (`schedule/context/schedule-context.tsx`) объединяет несколько подсостояний:

| Хук | Что хранит |
| --- | --- |
| `useWeekRange` | Текущая неделя (`from`, `to`), навигация по неделям, ограничители `±1 год` |
| `useSelectedEntity` | Текущая «точка зрения» — выбранная группа/преподаватель/аудитория + история выбранных |
| `useEditLessonControl` | Состояние модалки редактирования урока (открыт / закрыт, prefill) |
| `useLessonAtoms` | Массив `lessonAtoms` + `applyAtomsAppend / Remove / Replace` |
| `useLessonClipboard` | Буфер обмена (copy/paste пар) |
| `useScheduleHistory` | Undo/redo стек (см. ниже) |
| `useStatistics` | Производная статистика для правой панели (часы у преподавателя, плотность, ...) |
| `useWarningsCount` | Количество активных конфликтов |

Контекст экспортирует методы-фасады (`appendLessonAtoms`, `removeLessonAtoms`, `replaceLessonAtoms`, `selectEntity`, `undo`, `redo`), которые **записывают действие в историю** и затем применяют его. Это даёт единую точку для undo/redo: любое мутирующее действие пользователя автоматически записывается.

### 7.3. История (undo / redo)

`useScheduleHistory` (`context/use-schedule-history.ts`) реализует классический past/future стек с лимитом `HISTORY_LIMIT = 100`:

```
past: ScheduleHistoryEntry[]      ← применённые действия, готовые к undo
future: ScheduleHistoryEntry[]    ← откатанные, готовые к redo
```

Поддерживаются 4 типа записей:

- `atoms-append` — добавление атомов;
- `atoms-remove` — удаление;
- `atoms-replace` — замена (для редактирования урока);
- `entity-select` — переключение точки зрения.

Хук возвращает `recordHistory(body) → id`, `undo(expectedId?)`, `redo()`, флаги `canUndo / canRedo`. Уникальный `id` нужен для **toast-undo-flow**: после удаления показывается toast «Удалено. Отменить?», по нажатию вызывается `undo(id)` — если за это время пользователь успел совершить новое действие, undo ничего не сделает (id не совпадёт).

Toast-флоу синхронизируется через `pendingToastsRef` — список активных toast'ов, которые автоматически закрываются при любом следующем действии (`dismissPendingToasts`).

### 7.4. Сетка расписания

`ScheduleGrid` (`schedule/grid/grid.tsx`):

- 6 дней по горизонтали (понедельник — суббота), `N` слотов по вертикали (из `Timetable` вуза);
- адаптивный zoom через хук `useResponsiveZoom` — на узких экранах сетка масштабируется, чтобы помещалась;
- ячейка идентифицируется `cellKey(date, slotIndex)` — это ключ в `Map<string, Lesson>`, заполняемом из агрегации атомов;
- `LessonRow` рендерит одну строку слота, `<LessonCard>` — карточку урока;
- `MoveCursorOverlay` — слой для drag&drop пар между ячейками.

`ScheduleHeader` (`schedule/header/header.tsx`) даёт переключение недель, undo/redo (через `use-history-shortcuts.ts` — `Cmd+Z` / `Cmd+Shift+Z`), фильтры, кнопку «Опубликовать».

### 7.5. Поток публикации изменений (целевой, MVP)

К защите запланирован MVP-flow «сохранения и публикации»:

1. Пользователь редактирует расписание, изменения копятся в `lessonAtoms` (origin = `dirty` / `new` / `removed`).
2. По кнопке «Опубликовать» открывается **диалог сводки изменений** — список всех затронутых сущностей с дельтой («Группа ИВТ-21: +2 урока, -1 урок», ...).
3. На сервере вызывается валидация: проверяются коллизии (один преподаватель в двух местах одновременно, аудитория занята, ...) и бизнес-предупреждения.
4. При наличии конфликтов кнопка «Опубликовать» **заблокирована**, пользователь видит список конфликтов и устраняет их вручную.
5. После устранения конфликтов — публикация **одной транзакцией**: все изменения попадают пользователям мобильных приложений / просмотра расписания мгновенно.

**Чего в MVP сознательно нет:**
- черновиков расписания (один сотрудник = одна сессия изменений, до публикации);
- многопользовательской координации (если двое сотрудников редактируют одновременно — побеждает тот, кто опубликовал последним);
- A/B-веток расписания.

Это сознательное упрощение MVP, подаётся в ВКР как такой выбор (раздел 14, перспективы).

│   └── *-chart.tsx            — графики через recharts
└── dialog/                    — диалог создания/редактирования
```

**TanStack Table** даёт сортировку, фильтрацию, пагинацию из коробки. **Recharts** — для аналитических графиков на странице (например, распределение групп по факультетам, диаграмма типов аудиторий).

`/workspace/dashboard` — главная страница сотрудника: карточка вуза, ключевые показатели (число групп, преподавателей, заполненность расписания), changelog последних действий, графики `entities-chart`, `schedule-chart`.

## 9. Настройки (settings dialog)

`workspace/settings/` — **модальный диалог**, не отдельный URL. Подключается из шапки. Содержит подразделы (`nav.tsx`, `navigation.ts`):

- **General** — отображаемое имя пользователя, аватар (`general/avatar-picker.tsx` + `avatar-config.ts`), смена пароля;
- **University** — название/акроним/логотип/домен учебного заведения (только администратор);
- (планируются) — уведомления, безопасность, биллинг.

Загрузка аватара использует `FileReader` + canvas-resize до 256×256 перед отправкой на бэкенд.

## 10. Локализация и темизация

Используется **общий с публичной частью** `next-intl` (см. [web-public.md](web-public.md#6-локализация-next-intl)) — переводы сервиса лежат в `messages/{en,ru}/pages/{authorization,onboarding,workspace}.json`. Темизация — общая (`next-themes`).

## 11. Состояние, формы, валидация

- **Локальное состояние:** React `useState` + кастомные хуки (`useValidation`, `useSchedule`, `useOnboardingSteps`).
- **Глобальное состояние сервиса:** React Context (`ScheduleProvider`, `WorkspaceHeaderProvider`).
- **Формы:** контролируемые компоненты shadcn/ui (`Input`, `Field`, `Label`, `Checkbox`, `Select`, `Combobox`). Без React Hook Form / Formik — для текущего объёма достаточно собственного `useValidation`.
- **Валидация:** хук `useValidation(value, rules[])` с готовыми правилами (`onlyLatin`, `invalidEmail`, `required`, ...). На сервере — повторная валидация (zod-схемы, целевая реализация в Server Actions).
- **Уведомления:** `sonner` (toasts), `AlertDialog` (подтверждения), inline-errors (`field.tsx`).

## 12. UX-особенности

- **Горячие клавиши:** `⌘Z / Ctrl+Z` — undo, `⌘⇧Z / Ctrl+Shift+Z` — redo (`use-history-shortcuts.ts`), `⌘K / Ctrl+K` — глобальный поиск (cmdk).
- **Toast-with-undo** — при удалении показывается «Удалено. Отменить?», как в Gmail (см. раздел 7.3).
- **`localStorage`-стабильность**: сворачивание сайдбаров, выбранный вуз, история выбранных сущностей — сохраняются между сессиями.
- **Адаптивность**: на `<768px` оба сайдбара принудительно сворачиваются, редактор расписания горизонтально скроллится.
- **Skeleton-state**: на этапе загрузки таблиц / расписания показываются `<Skeleton>`-плейсхолдеры.

## 14. Ограничения текущей реализации (для главы 3.5.5 и заключения)

1. **Нет автотестов** (симметрично всему стеку).
2. **Нет CI/CD** для веб-проекта (push → SSH-перезапуск контейнера).
3. **Нет real-time-сотрудничества** — если двое сотрудников редактируют одновременно, побеждает тот, кто опубликует последним. Многопользовательская координация (WebSocket / CRDT / operational transforms) — задача дальнейшего развития.
4. **Нет черновиков и веток расписания** — изменения существуют только в сессии до публикации.
5. **Нет аудит-лога действий пользователей** — changelog есть для бизнес-сущностей, но не для административных действий сотрудников.
6. **Нет push / email-уведомлений** сотрудникам о событиях (новый комментарий, конфликт, согласование).
7. **Нет интеграции с внешними системами вуза** (1C, LDAP/SSO, импорт расписания из CSV/iCal).

## 16. Кросс-платформенные наблюдения

Сервис администрирования — **единственный продукт экосистемы, который не имеет аналога в мобильных клиентах**. Это сознательное продуктовое решение: задачи методиста (составление расписания на неделю в визуальной сетке) принципиально лучше решаются на большом экране с клавиатурой. Мобильное приложение для администратора **не планируется**.

Архитектурно сервис **берёт от мобильных клиентов**:

- единый домен (`University`, `StudentGroup`, `Tutor`, `Classroom`) — расширенный административными сущностями (`Faculty`, `Department`, `Employee`, `Building`, `LessonAtom`);
- pattern «доменная модель → DTO → Mapper» (хотя для админ-сервиса больше работы делается на RSC, и часть DTO-маппинга уходит на серверную сторону);
- кросс-платформенная identification модель (роли, X-Installation-ID для аналитики, общий клиентский ключ платформы — теперь скрытый за серверным прокси).
