## Поток данных в парсинговой модели

1. Клиент запрашивает расписание на бэкенд.
2. Бэкенд берёт расписание с сайта вуза, преобразует в единый формат, возвращает клиенту.
3. Клиент кэширует локально и показывает пользователю.

## Поток данных в SaaS-модели (целевая)

1. Методист вуза вносит изменения в веб-сервисе администрирования.
2. Backend выполняет валидацию (проверка коллизий по преподавателю, аудитории, группе).
3. Запись сохраняется в БД.
4. Обновлённое расписание доставляется во все клиентские приложения (через pull или push — см. [backend-contract.md](backend-contract.md)).

## Контракт между клиентом и бэком

Минимальное, что известно со стороны клиентов:
- Запросы возвращают расписание в едином формате (JSON).
- Уникальный UUID, генерируемый при первом запуске клиента, отсылается с каждым запросом — это используется для подсчёта уникальных и новых пользователей.
