Перспектива объединения: парсинговый слой встроится в платформенный контур как один из источников данных. Внешне для iOS/Android/Web-public ничего не меняется — они продолжают ходить на `api.vcourse.app`. Внутри запрос расписания вуза смотрит, есть ли расписание в собственном сервисе; если нет — отдаёт результат парсинга.

## Контракт клиент → парсинговый контур (`api.vcourse.app`)

Одинаков для iOS, Android и Web-public:

- **Формат:** REST + JSON.
- **Заголовки:**
  - `Authorization: Bearer <UUID>` — единый клиентский ключ платформы (не персональная авторизация, идентифицирует факт легитимного клиента «ВКурсе»);
  - `X-API-Version: 2`;
  - `X-Installation-ID` — UUID конкретной установки приложения (только iOS/Android);
  - `User-Agent` — собирается клиентом из имени, версии, билда, ОС, модели устройства.
- **Окно загрузки расписания:** ±1 год от текущей даты, формат дат `yyyy-MM-dd`.
- **Эндпойнты:** `GET /universities`, `GET /timetable/{universityId}`, `GET /schedule/{universityId}/{group|tutor|classroom}/{entityId}?since=&until=`, `GET /schedule/.../count`, и справочники.

**Web как BFF.** Клиентский Bearer-токен платформы **не передаётся в браузер**. Все запросы из браузера идут на собственные Next.js Route Handlers (`/api/vcourse/[...path]`, `/api/static/[...path]`), которые подставляют `Authorization` из server-side переменной окружения `VC_AUTH_TOKEN` и кэшируют ответы бэкенда (1 час на API, 24 часа на статику). Это устраняет риск утечки токена через DevTools и снижает нагрузку.

## Контракт web-admin → платформенный контур (`platform.vcourse.app`)

- **Формат:** REST + JSON.
- **Спецификация:** OpenAPI + Swagger (на момент ревью публично — только auth-сервис).
- **Аутентификация:** OAuth 2.0 Resource Owner Password Credentials Flow. После логина клиент получает пару JWT-токенов: access (TTL 5 мин) + refresh (TTL 90 дней). Токены хранятся в безопасном хранилище браузера.
- **Авторизация:** в access-token приходят `authorities` (глобально: `ROLE_USER`/`ROLE_ADMIN`) и `orgs` (per-org-роли). В MVP пользователь привязан только к одной организации.
- **Пагинация:** Page-Based (`page`+`size`) для списков сущностей, Time-Based (`since`+`until`) для расписания.
- **Версионирование:** заголовок `X-API-Version` (как и в парсинговом контуре).

## Поток данных

### Просмотр расписания (студент / преподаватель)

1. Клиент запрашивает расписание на `api.vcourse.app`.
2. Бэкенд либо отдаёт расписание из собственной БД (если вуз подключён к собственному сервису), либо парсит сайт вуза и преобразует в единый формат.
3. Клиент кэширует локально (SwiftData / Room / Next.js ISR) и показывает пользователю.

### Управление расписанием (сотрудник вуза)

1. Сотрудник авторизуется в web-admin → получает JWT-пару от auth-сервиса.
2. Web-admin читает справочники и расписание через RSC-запросы к Core-сервису на `platform.vcourse.app`.
3. Изменения копятся локально в `ScheduleProvider` (атомарная модель `LessonAtom` + история действий, см. [clients/web-admin.md](clients/web-admin.md)).
4. По кнопке «Опубликовать» Server Action отправляет изменения в Core-сервис; тот выполняет бизнес-валидацию (коллизии преподаватель/аудитория/группа) поверх constraints PostgreSQL.
5. Опубликованное расписание становится доступно через `api.vcourse.app` всем клиентам просмотра.

Подробности — в [backend-contract.md](backend-contract.md) и [clients/web-admin.md](clients/web-admin.md).
